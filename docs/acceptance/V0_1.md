# v0.1 acceptance contract

This checklist maps `VISION.md`'s definition of success to concrete evidence. A
passing build alone is not sufficient.

## Automated gates

- `npm run verify` passes formatting, strict type checking, unit tests, and
  integration tests across all workspaces.
- `npm run build` produces the engine, Node runner, and desktop application.
- Engine tests prove structural parsing, actionable semantic validation, DAG
  planning, parallel scheduling, artifact routing, failure propagation, and
  cancellation.
- Runner integration tests prove direct execution, explicit shell mode, stdin,
  environment and working-directory handling, stdout/stderr capture, non-zero
  exits, missing executables, JSON output failures, filesystem-reference checks,
  and abort behavior.

## Desktop acceptance

Using the packaged or development desktop application on macOS:

1. Create three process blocks and arrange them on the canvas.
2. Configure two independent source blocks and connect both to a downstream
   block.
3. Verify that a cycle and an incompatible port connection are rejected with
   actionable messages.
4. Review the exact executable, arguments, working directory, environment
   bindings, and shell status before execution.
5. Run the workflow and observe the source blocks running independently before
   the downstream block.
6. Inspect resolved inputs, text/JSON/filesystem artifacts, stdout, stderr, exit
   result, and timing for each block.
7. Run a long-lived process, cancel it, and verify active and downstream states
   become cancelled or skipped rather than succeeded. If operating-system
   termination itself fails, an actionable failed state is required because the
   process may still be running.
8. Run workflows that demonstrate a missing executable, malformed JSON output,
   inaccessible filesystem reference, and non-zero exit; verify each failure is
   distinguishable and actionable.
9. Save the workflow, close or replace it, reload it from disk, and verify its
   structure, configuration, connections, and layout are restored.
10. Confirm saved workflow JSON contains environment-variable references but no
    resolved host secret values.

## Completion evidence

Recorded on 2026-07-09 on macOS:

- `npm run verify` passes strict formatting and type checks plus 58 tests: 26
  engine, 17 Node runner, and 15 desktop tests.
- `npm run build` produces the engine, Node runner, Electron main/preload, and
  renderer bundles.
- `npm run desktop:smoke` launches the production Electron build and proves the
  sandboxed preload bridge, visual creation of three process blocks, authority
  review, explicit PATH reference, successful execution, run-detail inspection,
  cancellation, a typed missing-executable failure, and a full pointer drag that
  keeps both the canvas node and minimap node mounted for every movement frame.
  It also captures `docs/acceptance/V0_1_EDITOR.png` and
  `docs/acceptance/V0_1_DESKTOP.png`.
- `npm run workflow:check -- examples/workflows/hello-report.vorchestra.json`
  validates the portable two-block example and its execution layers.
- `npm run workflow:run -- examples/workflows/hello-report.vorchestra.json`
  executes Node twice and routes JSON through downstream stdin to produce
  `Hello from Vorchestra!`.
- Engine tests cover compatible/incompatible connections, cycle rejection,
  parallel branches, every artifact kind, routing, skips, typed failures,
  cancellation precedence, and observer isolation.
- Node-runner tests cover stdin/stdout/stderr, explicit environment and cwd,
  direct/shell execution, non-zero and signal exits, JSON and filesystem output,
  missing executables, inaccessible paths, cancellation races, descendant
  cleanup, and SIGKILL escalation.
- Desktop persistence tests prove atomic save/load round trips for workflow
  structure and layout, preserve host-variable references, exclude resolved PATH
  values, and leave an existing file intact when replacement data is invalid.

Packaged Windows and Linux desktop behavior remains outside v0.1 acceptance.
